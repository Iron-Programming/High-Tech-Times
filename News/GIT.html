GIT & Github: The best software developer tools.

Introduction to GIT & Github

If you asked me what tool I thought was the most useful for code developers, I would undoubtedly say GIT & Github. Now obviously you need a good coding editor as well, but that is kind of already assummed.
GIT and Github are extremely helpful because of two reasons. First, they make changing your code project extremely easy. Accidentally made a change you didn't like? No problem. Just look through your project history and find the version of the code you are looking for. The second reason that makes these tools so great is that because they are built for specifically for project management, they work amazing for teams.

Now GIT is the actual tool, and is run on your local computer. GIT works great, but it takes some time to learn the commands before you can use it. This is where the beauty of Github comes in. Github is an webpage (online) that lets you use the functionality of GIT from any of your devices--this makes collaborating with others even more easy! Finally, as it is not as convenient to actually do the coding process online, Github has an desktop app called Github Desktop which lets you connect to your Github account on your local device, and then you can code in your Coding Editor (or Integrated Development Environment) as you need.
In this article I will be showing you the underlying concepts used in GIT, but I will show you how to implement it via Github and Github Desktop, as they are more user-friendly than the command-line interface we have to deal with when using straight-out GIT. Enough talk, let's get started!

Underlying concepts of GIT

So GIT is a version control system, which basically means that it tracks changes to our project files (in this case, coding files such as HTML, CSS, and JavaScript). GIT is the most popular version control system out there, and was originally developed by Linus Torvalds in 2005 to make developing the Linux kernel easier.
The first concept crucial to GIT's functionality is that of commits. A commit is best thought of as a "save point": just like how you can restart at your save point in a game if you lose, you can likewise restart back to previous versions of your code if you accidentally mess up. This is extremely useful if you ever need to access code that you wrote earlier for the project but that you replaced. This way, you don't have to store important bits of code you might use later, GIT will keep track of everything for you!
The other term you will hear a lot when working with GIT is a repository, often abbreviated as repo. A GIT repo is the directory holding your files, and it is what manages and keep tracks of your files, as you make commits and whatnot.

In GIT, you would have commands to create files, add them to the "staging area", commit, revert back to previous commits, and so on. But because we won't be working with GIT directly, we won't be going over those things right now. Instead, we will now take a short look at how Github works before showing it in action.
So GIT can be used without internet access, but one of the things that really make GIT shines is collaboration. The best way to do this is to store a GIT repository online, and then each of the code developers can make a clone of that repository and work on it at their desktop. The code developers can pull the code from the online repository (called the origin) and they can push it back when they have made changes. Github is a great place to store your repository, but it is not the only website offering this service. Another two common websites offering this service include GitLab and BitBucket.

Getting started with Github

Github is owned by Microsoft, and is the most popular place to store our remote repositories. To get started create an account.

  • At the link given (above), provide your email, password, and username.
  • Choose if you want to receive product updates.
  • Solve Puzzle to verify that you are a human.
  • Create your account!
  • To continue from there, you will need to verify your account by going to the email you attached to your account and get the access code.
  • You can then continue with a free account. If you are a student, I would definitely recommend checking out software development pack for students later on, but for now, just continue with a free account.

Like all apps, Github will take some time to get used to. You can explore Github and customize your account as you enter your journey into Github. For now, we will want to pay attention to your Github Repositories section, which will currently be empty.
In this example, I will be showing you how to create a Github Repo for an webpage. Another nice feature about Github is that it has a feature (Github Pages) which lets you view your webpage in the browser. This way, a group of web developers can make sure they are all on the same page, literally!
But anyway, for now we just need to create a repository, as shown in the image to the right. When we create a new repository, we will need to provide the following information.

  • Repository name (required)
  • Description (optional)
  • Public vs Private
  • You can choose whether to add a README, .gitignore, and license files.

If you plan to collaborate, make sure to configure your repository as public. The README file is simply a description of your repository. The License File determines how users can use your code. I like to use a more liberal license like MIT.

From here, you can manage your project. You can click on "Add file" to either create a new file directly on Github or upload files from your dekstop. To great a directory on Github directly, you can name your file /somedirectory/filename, where is "somedirectory" doesn't exist it will be created.

Earlier I said that we would be creating a webpage with this. First, we need to create an index.html file, which is the file that Github Pages starts the webpage at. Below is a sample file for Hello World. 

<!DOCTYPE html>
<html>
  <head>
    <title>Hello World!</title>
  <head>
  <body>
    <p>Hello World, this is my first Github repo!</p>
  </body>
</html>

Notice that to add the changes to the Github repo you must "commit" the file, adding a description for what you changed.
After we added index.html, we can go to Settings > Github Pages and configure the appropriate settings.
The only thing you need to do is select the "None" dropdown box and choose to build GithubPages from the main branch.
It will take up to 5 minutes for Github to recognize the newly configured pages. After that, you should have a brand new website!

Using Github Desktop

Github Desktop gets the best of both worlds: the amazing collaboration features from Github, and the convenient coding on your desktop. Don't forget of course, that we are using GIT to keep track of our files.
Before preceding, make sure that you have a Coding Editor that you can use. I would recommend Atom or VisualStudio, both of which are available on either Mac or Windows (VisualStudioCode for Mac).
First things first, we need to download the Github Desktop app. Once we have done that, we are ready to roll!

Okay, so once you are in the Github Desktop app, you need to connect to your Github account. You should see an opening screen welcoming you and you should be able to conent your account from there. Although I usually use Github Desktop on a Mac, it has a nicer interface with more options on Windows.
After you are connected, you can clone a repository from Github by going to File > Clone Repository (you can optionally create a new one).
Once you have cloned the repository, you should have a interface looking something like the image you see to the right/above.

At this point, we have everything set up, and the only thing to do now is to code! The process is not too complicated. To start, we should always fetch the origin to make sure we are up to date with the online Github repository. Then we can open up our repository on our desktop using our default coding editor (as shown in image), which you will need to set first time.
Now, whenever you make changes to your local repository, such as adding new files are changing existing ones, you are not utilizing GIT yet. To make another 'save point' we need to commit our changes to the main branch. We can do this by clicking in the Github Desktop interface and then adding a commit.
The area showing your changed, added, or deleted files is the graphical representation of what is called the "Staging Area" in GIT. We can remove which files we want to be in the staging area by clicking the check mark. All the checked items are in the staging area and thus will be committed whenever we make a commit.
As see in the screenshot above/left, I was making a commit to this article. The "green +" indicates a newly created file, and a "yellow dot" indicates a changed file.
Once you have made one of more commits on your local repository, the option for "Fetch origin" should change to "Push origin". Now you can push your local repository up to the origin (the Github repository).

The last big component of GIT is branches, and this is where GIT can really shine with collabotive work. To create a branch, click on "Current Branch" and click "Add new".
Whenever you work on a branch, you are not making changes to the main branch. Thus you won't accidentally ruin the main project (website in our case). This is especially handy for web developers who work to maintain their website. They can work on another branch besides main and once they are sure they have made the correct changes, then they can merge the branch back into the main/master branch.

Summary

And that is the very basics of GIT, Github, and Github Desktop. There is a lot more to GIT and Github's functionality, especially when it comes to branching and making pull requests. Feel free to send us a request if you want us to make more tutorials about GIT and Github!
